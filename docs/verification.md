 # 默认结果验证流程（网页 vs xlsx）
 
 目标：在不打开 Excel 的情况下，快速验证网页在“默认配置”下的计算结果与 xlsx 缓存值近似一致；若不一致，能够快速定位到具体单元格/公式/函数差异。
 
 ## 1. 验证原理
 
 xlsx 文件里很多公式单元格带有“上次保存时的缓存结果值”。网页在浏览器端用公式引擎重算后，可把：
 
 - 网页计算值（引擎输出）
 - xlsx 缓存值（SheetJS 读出的 cell.v）
 
 做近似比较，从而判断“计算是否基本对齐”。这能快速发现：函数不支持、公式解析差异、数据读取错误、扩展列组导致引用范围不对等问题。
 
 ## 2. 一键验证（推荐）
 
 打开主界面时在 URL 加上参数：
 
 - `/cloud-team?verify=1`
 
 页面右侧“结果区”下方会显示验证报告：
 - 通过：显示“验证通过：与xlsx缓存结果近似一致”
 - 不通过：输出不一致的条目（包含 cached / computed / addr）
 
 当前默认检查项（可按需要扩充）：
 - 云配队!AD5（累计伤害）
 - 云配队!AD7（DPS）
 - 云配队!AG4~AG7（4角色贡献伤害）
 
 容差策略：
 - 数值：相对误差 0.1%（|a-b| <= max(1e-6, |a|*1e-3)）
 - 非数值：字符串全等
 
 ## 3. 常见不一致原因与溯源路径
 
 ### 3.1 公式函数不支持/名称不同
 
 本表使用的函数集合很小（17个），唯一带前缀的是：
 - `_xlfn._xlws.FILTER`
 
 网页侧会把它预处理成 `FILTER` 再交给公式引擎。
 
 若验证失败且涉及 FILTER 相关输出：
 - 先检查公式引擎是否支持 FILTER 与数组溢出
 - 再检查 FILTER 的输入范围与条件是否正确读入
 
 ### 3.2 动作位扩展导致引用范围不覆盖新列
 
 当动作位数量从 20 扩展到更大时：
 - 云配队 会在 AC 之后插入新列承载动作/buff
 - 计算 sheet 里所有引用 `云配队!$Jx:$ACx` 的公式需要被重写为 `云配队!$Jx:$<新右边界列>x`
 
 如果扩展后结果明显不对：
 - 优先检查：计算 sheet 公式是否已完成范围重写
 - 再检查：新动作位/新 buff 位是否写入正确的列
 
 ### 3.3 下拉选项来源变化（动作/弧盘/增益）
 
 网页动作下拉来自 “动作”sheet 的 A/B 列映射；弧盘下拉按 “弧盘”sheet 的 F 列适配过滤；buff 下拉来自 “增益”sheet 固定区间。
 
 若表更新后出现：
 - 下拉缺项/错项
 
 先对照 [xlsx-bindings.md](file:///Users/bytedance/projects/nte_quick_team_up/docs/xlsx-bindings.md) 的数据源列/范围是否变化。
 
 ## 4. 浏览器缓存说明
 
网页会对 xlsx 做持久缓存，避免多次拉取同一个文件：

- 优先：CacheStorage（cacheName：`cloud-xlsx-v1`）
- 兼容降级：若运行环境不支持 CacheStorage（例如 `caches is not defined`），使用 IndexedDB 做持久缓存（DB：`cloud-xlsx-cache`）
 
 如果你更新了仓库内的 xlsx 文件但浏览器仍使用旧文件：
 - 改一下 cacheName（例如 `cloud-xlsx-v2`）或
 - 在浏览器 DevTools 里清理站点缓存/存储

另外，网页还会把“可编辑状态”（JSON）存入 localStorage（例如角色选择、动作/buff 配置、配装页输入）。如果出现“我改了代码但页面还是旧表现/旧数据”的情况，优先检查：
- 是否命中了旧的静态资源（可通过模板里的 `?v=...` 强制刷新）
- 是否需要清理站点 localStorage（或升级 STORAGE_KEY）
